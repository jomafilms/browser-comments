# External Developer Setup

Pull feedback tickets into your Claude Code (or any AI agent) workflow using a project-scoped token.

## 1. Get your project token

Your project admin will provide a **project token** — a 32-character string that gives you access to tickets for your specific project only. You will not see tickets from other projects.

## 2. Add token to your project

Add to your project's `.env.local`:

```
BROWSER_COMMENTS_TOKEN=your_project_token_here
BROWSER_COMMENTS_API=https://your-instance.vercel.app
```

## 3. Choose your integration

### Option A: MCP Server (recommended for Claude Code)

Add once to your global Claude Code config (`~/.claude/settings.json`):

```json
{
  "mcpServers": {
    "browser-comments": {
      "command": "npx",
      "args": ["-y", "@jomafilms/browser-comments-mcp"]
    }
  }
}
```

The MCP server reads `BROWSER_COMMENTS_TOKEN` and `BROWSER_COMMENTS_API` from each project's `.env.local` (step 2 above). You only configure the MCP server once globally — every project with a token in `.env.local` will work.

This gives your Claude Code instance tools: `list_tickets`, `show_ticket`, `resolve_ticket`, `reopen_ticket`, and `assign_ticket`.

Available tools:

| Tool | Description |
|------|-------------|
| `list_tickets` | List tickets with optional filters (status, priority, assignee, section) |
| `show_ticket` | Show details for a single ticket by number |
| `resolve_ticket` | Mark a ticket as resolved, optionally with a note |
| `reopen_ticket` | Reopen a previously resolved ticket |
| `assign_ticket` | Assign a ticket to a team member |

### Option B: CLI tool

Install the CLI globally:

```bash
npm i -g browser-comments-cli
```

Then from any project directory with `.env.local` containing your token:

```bash
# List open tickets
browser-comments list --status open

# Show a specific ticket
browser-comments show 3

# Resolve a ticket with a note
browser-comments resolve 3 --note "Fixed in commit abc123"

# Assign a ticket
browser-comments assign 3 --to "Dev Name"
```

Claude Code can run these via `! browser-comments list` in the prompt.

### Option C: Direct API

All endpoints accept your token as a query param or Bearer header:

```bash
# List tickets
curl "https://your-instance.vercel.app/api/comments?token=YOUR_TOKEN&excludeImages=true"

# Update a ticket
curl -X PATCH "https://your-instance.vercel.app/api/comments/42" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"status": "resolved", "note": "Fixed"}'
```

## 4. Add the feedback widget (optional)

If you want testers to submit visual feedback on your dev/staging site, add this script tag:

```html
<script
  src="https://your-instance.vercel.app/widget.js"
  data-key="YOUR_WIDGET_KEY"
></script>
```

Your admin will provide the widget key separately. Optional attributes:

| Attribute | Default | Description |
|-----------|---------|-------------|
| `data-user-name` | _(prompt)_ | Pre-fill submitter name |
| `data-position` | `bottom-right` | Button position |
| `data-color` | `#2563eb` | Button/UI accent color |

## Security notes

- **Project tokens** scope access to a single project's tickets. You cannot see or modify other projects.
- **Client tokens** (held by admins) grant access to all projects under a client. You should never need one.
- Tokens can be revoked/regenerated by the admin at any time.
- Never commit tokens to version control — keep them in `.env.local` (which should be in `.gitignore`).
